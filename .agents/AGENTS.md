# Homelab Architectural Rules

The following rules dictate how the homelab repository is structured and managed. Agents must strictly adhere to these guidelines when making modifications.

## 1. Decoupling Infrastructure from Source Code
- The `nodes/` directory is strictly for Infrastructure as Code (Docker Compose, systemd configs, deployment scripts).
- Custom application source code (web apps, data pipelines) MUST live in the `projects/` directory. They should never be mixed with node configurations.

## 2. Secrets Management (Nested Private Repositories)
- Passwords, API keys, and VPN configurations (`.env`, `*.conf`) are NOT encrypted with SOPS. 
- Instead, sensitive directories (e.g., `nodes/homelab-95/secrets/`) are added to the root `.gitignore`.
- These directories act as standalone nested Git repositories that push to completely separate, private GitHub repositories for automated backup.

## 3. Safe Information
- Internal RFC1918 IP addresses (e.g., `192.168.50.x`) and standard host usernames (e.g., `k`, `root`) are considered safe and SHOULD be tracked in the public repository (e.g., in `shared/ssh/config` or markdown documentation).

## 4. Handling Nested Projects & Submodules
- When adding standalone projects to the repository (e.g., `projects/airflow-dags` which has its own `.git` folder), you MUST explicitly add the folder path to the root `.gitignore`. 
- We do not use formal Git Submodules (`.gitmodules`), as ignoring the folder creates a cleaner boundary and prevents detached HEAD errors.

## 5. Ephemeral Build Artifacts
- Tooling dependencies and build outputs (e.g., `dbt_packages/`, `target/`, `node_modules/`, `logs/`) MUST be ignored via `.gitignore`.
- Unlike secrets or private code, these artifacts do NOT require private repositories or backups, as they can be dynamically regenerated by running the respective tools.
